# Adaptive v2.2 Final Report

Status: exploratory preregistered study complete (16/16 trials, 64/64 arms).

Protocol v2.2 tested four ways of giving Codex repository context: no Palace,
Route-only, always-on Full Palace, and Adaptive Palace. The protocol and all
trial assignments were frozen at tag `protocol-v2.2.0` before any v2.2 agent
outcome existed.

## Executive Result

Adaptive Palace reliably produced a smaller Palace-owned payload than Full
Palace. It did not establish better correctness, lower end-to-end Token use,
or faster execution than the no-Palace Control. The strongest supported value
in this study is accurate, auditable context routing and guarded project
history, not a universal efficiency claim.

Every trial ran sequentially with a 15-second cooldown on the frozen

used warm indexes and two forced cold rebuilds. The four Williams sequences
placed every arm in every order position once within each scenario.

## Primary Correctness Comparison

Adaptive versus Full Palace was the frozen v2 primary comparison. Both arms
succeeded in all 16 valid pairs. Every scenario therefore had a 0 percentage-
point success difference, exact McNemar p = 1, and Holm-adjusted p = 1.

This tie is evidence of non-degradation inside these fixtures, not evidence
that project memory improves correctness. The useful- and stale-memory tasks
were solvable from visible code and tests by all treatments.

## Mechanism Findings

1. **Routing coverage was strong.** All 48 Palace arms had route recall of
   1.0 for the scenario ground truth. Median precision was 0.5, 0.8, 0.5, and
   2/3 across the four scenarios.
2. **Adaptive mode selection was stable.** It chose `route-lite` 4/4 times for
   the one-file task, `full-palace` 8/8 times for cross-stack and useful-memory
   tasks, and `guarded-memory-palace` 4/4 times for stale memory.
3. **Useful-memory fidelity failed.** Full delivered both seeded Aurora
   notices 4/4 times. Adaptive delivered neither, reported zero memory items
   and guardrails, and emitted the same 2,450-byte context in all four trials.
   Its apparent savings are not an equal-content memory comparison.
4. **Stale-memory guardrails worked.** Adaptive delivered both stale records
   plus two current-code-first guardrails 4/4 times. No treatment adopted the
   stale edit, so comparative benefit remains unproven.
5. **Smaller routing output did not guarantee a shorter agent run.** The
   agent could spend saved context on additional inspection. Palace payload
   and Codex end-to-end usage must remain separate metrics.

## Supported And Unsupported Claims

Supported by this exploratory study:

- Adaptive reduces Palace-owned payload relative to always-on Full Palace.
- The router covered every required file in these fixtures.
- Guarded stale-memory mode consistently attached explicit safety language.
- All four treatments completed these fixtures correctly and safely.

Not supported by this study:

- Vertex Palace universally reduces Codex Token use or wall time.
- Vertex Palace improves correctness over ordinary Codex on these scenarios.
- Adaptive preserves useful Full-Palace memory in Vertex Palace 0.2.1.
- A smaller route or context pack by itself predicts lower end-to-end cost.

## Product Recommendations

1. **Fix memory parity before the next release claim.** When Adaptive selects
   `full-palace`, it must either include the same relevant memory as Full or
   explicitly report why each memory item was excluded. Add a regression test
   for the Aurora omission before publishing a new package version.
2. **Optimize for end-to-end behavior, not payload alone.** Add mode features
   that reduce follow-up exploration: one authoritative primary file, explicit
   dependency coverage, conflict summaries, and a stop condition after tests
   confirm the routed hypothesis.
3. **Permit true bypass for trivial work.** The small-local negative control
   showed higher Adaptive-versus-Control reported tokens. High-confidence
   one-file tasks with no relevant memory should be allowed to return a tiny
   pointer or bypass Palace context entirely.
4. **Expose treatment-fidelity telemetry.** Record selected mode, candidate
   memory count, included and excluded memory ids, guardrails, index freshness,
   payload bytes, and route coverage in machine-readable output.
5. **Run a fresh confirmatory protocol after fixes.** Use a new package
   version and frozen tag. Include larger real repositories, hidden memory-
   dependent traps, repetitions across models and machines, and enough pairs
   for useful precision. Do not reuse v2.2 outcomes as post-fix evidence.

## Limitations

- Four pairs per scenario produce coarse, exploratory intervals.
- Provider-side cache state, service load, and network latency were not
  controlled; wall time is secondary.
- Results use one model build, one Codex CLI build, one OS, and deterministic
  synthetic repositories. Generalization to other environments is unknown.
- Cumulative transcript Token counters are not an API billing statement.
- Secondary efficiency contrasts were not multiplicity-adjusted.
- The memory scenarios did not force a correctness difference, and the useful
  memory treatment had a discovered fidelity defect.

Raw transcripts remain local because they contain private paths and session
metadata. The public source of truth is the checksum-audited evidence under
`results/adaptive-pilot-v2.2/`, plus the generated `analysis.json`.

## 简体中文结论

v2.2 已完成 16 个 trial、64 个 Arm，全部有效、正确、通过公开测试与隐藏
Oracle。Adaptive 相对 Full Palace 稳定缩小 Palace 自身输出，整体中位数少
898.5 bytes；但端到端 Token、调用与时间的区间都跨越 0。相对完全不使用
Palace 的 Control，Adaptive 的工具调用中位数多 4.5 次，reported tokens 与
时间的中位数也更高。因此目前不能说 Vertex Palace 普遍省 Token 或更快。

真正被验证的价值是：路线覆盖完整、上下文可审计、过期记忆会附带防护语。
同时研究也抓到一个重要缺陷：v0.2.1 的 Adaptive 在 useful-memory 场景四次
遗漏已播种的记忆。下一步应先修复记忆一致性和额外探索成本，再以新版本、
新冻结协议做确认性实验。
